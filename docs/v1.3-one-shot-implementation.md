## Product decision

This release strengthens SourceCapsule's specific advantage: trustworthy, AI-ready, durable
capture of one X post, thread, or Article. It is not a general bookmark manager and should not
chase XClipper's batch-export surface area.

Target release: **v1.3.0 — Chrome parity, trustworthy capture, and first-use polish.**

The complete release includes:

1. MV3 extension parity and reliability repairs
2. A functional extension toolbar popup with accessible settings
3. A persistent, accurate capture receipt
4. Poll preservation across every output
5. A hosted-share acquisition footer that never appears in private local archives
6. Automated and manual validation material

## Non-negotiable engineering constraints

- Keep the userscript source as the production artifact; no bundler for `sourcecapsule.user.js`.
- Preserve the model contract: X DOM/network data becomes plain model objects before rendering.
- Keep every new X selector in the top-level `CONFIG.selectors` block.
- Preserve the `typeof module` export guard used by Node tests.
- Do not introduce an agent framework, backend account system, analytics, telemetry, or new paid
  service.
- Local capture remains primary. Sharing must remain explicit, confirmed, expiring, and secondary.
- Never claim media or text was captured unless the model/manifest proves it.
- Long operations must continue printing/updating live progress and must retain timeouts.
- Minimize extension permissions and keep development-only hosts out of Store packages.
- Maintain backward compatibility for existing Tampermonkey/Violentmonkey users.

## 1. MV3 extension parity and reliability

### 1.1 Repair share-service access

The source manifest allows the production Worker, but `extension-src/background.js` currently
rejects it at runtime. Repair this mismatch.

Required behavior:

- Production runtime allowlist includes
  `sourcecapsule-share.wolfgang-aura.workers.dev`.
- Production manifest and runtime allowlists stay synchronized.
- Remove the unused aspirational `share.sourcecapsule.app` permission and runtime allowance unless
  it is actually registered and used by the time this work executes.
- Production build excludes `localhost` and `127.0.0.1` permissions.
- Development build may include localhost permissions through an explicit build flag or separate
  development manifest transformation.
- Add automated tests for allowed production requests, rejected unknown hosts, and production
  packages containing no development hosts.
- Verify that `Share with AI` works from the generated MV3 extension against the configured
  production Worker, without changing userscript behavior.

### 1.2 Replace fragile inline bridge injection for MV3

The userscript currently injects an inline page script to observe X's already-fetched GraphQL
responses. That path may be blocked by page CSP when running as an extension.

Implement a static MV3 page-world bridge:

- Add an extension-owned bridge file loaded at `document_start` with manifest
  `"world": "MAIN"`.
- The bridge must install the minimum fetch/XHR observation needed by the existing passive network
  capture pipeline.
- Send captured payloads into the isolated SourceCapsule content script through a narrowly scoped,
  validated event contract.
- Validate event origin/shape, cap payload sizes, and never execute payload content.
- Prevent duplicate installation and duplicate event handling.
- Extension mode uses the static MAIN-world bridge; userscript mode retains its existing compatible
  injection mechanism.
- Confirm that passive long-form Note recovery and video candidate discovery work in extension
  mode.
- Add unit coverage around event validation, payload caps, duplicate suppression, and dispatch to
  the existing `handleNetworkCapturePayload` path.

### 1.3 Enforce one release version

The baseline is inconsistent:

- `package.json`: `1.2.2`
- userscript metadata: `1.2.2`
- internal `VERSION`: `1.2.1`
- source extension manifest: `1.2.0`

For v1.3.0:

- Align package, userscript metadata, internal exporter constant, and built extension version.
- It is acceptable for the source extension manifest to contain a clearly documented placeholder
  only if the build overwrites it and the audit validates the built output.
- Add or extend a release-audit test that fails on version drift.
- Confirm HTML generator metadata, Markdown provenance, debug manifest, archive manifest, logs, and
  extension package all report `1.3.0`.

## 2. Functional toolbar popup and extension settings

The current toolbar action has no popup, and `compat.js` turns userscript manager menu commands
into no-ops. Therefore extension users cannot reach important preferences.

Build a small MV3 popup using plain HTML/CSS/JavaScript.

### Popup states

- On a supported X page: `SourceCapsule is ready on this page.`
- Off X: explain that SourceCapsule runs on X and provide an **Open X** button.
- If the current X tab needs refreshing after install/update, say so explicitly.
- Show the installed SourceCapsule version.

### Popup controls

- Library layout: `By date` or `Flat`
- Library contents: `Full archive` or `Lean Markdown + media`
- Floating button: on/off
- Change export folder
- Help/source link
- Privacy link

### Architecture

- Do not duplicate extraction or export logic in the popup.
- Add a narrow content-script controller/message API that reads and writes the existing preference
  model.
- Keep the content script as the source of truth for preferences.
- Validate every incoming message and return friendly failures when no SourceCapsule content script
  is available.
- Avoid adding broad `tabs` access if the same result can be achieved with `activeTab`, existing
  host access, or messaging.
- Folder selection must not fail silently. If Chrome does not preserve the required user gesture
  through popup-to-content messaging, render an in-page prompt/button and require the user to click
  it directly before opening the directory picker.
- Userscript-manager menu commands must continue working unchanged.
- Add tests for popup/controller get-state, preference changes, unsupported-tab behavior, and
  folder-picker fallback messaging.

The popup is functional UI. Store artwork, promotional screenshots, and final branded icon assets
remain the user's Store-listing track; do not block the code on them.

## 3. Persistent capture receipt

The present success state is usually a short-lived toast containing a path or filename. Replace
that with a compact persistent receipt after successful local saves and downloads.

### Receipt content

- Clear success heading
- Saved folder/file location when available
- Posts captured
- Images captured and missing
- Videos preserved offline
- Video posters captured
- Incomplete media count
- Recovered long-form Notes count
- Overall status:
  - `Complete — no missing media detected`, or
  - a clear warning summary with a way to expand details

### Receipt actions

- **Copy for AI** — copy the already-built clean Markdown
- **Create share link** — enter the existing confirmation/expiry flow; never upload immediately
- **Done**

### Receipt rules

- Derive all counts from the existing archive statistics/manifest pipeline. Do not create a second
  competing truth calculation.
- Keep the successful local-save result visible even if a subsequent share attempt fails.
- Do not show a second competing modal for share success; integrate or coordinate with the existing
  share-result dialog cleanly.
- Use semantic dialog markup, keyboard focus management, Escape/close handling, and readable
  light/dark styles.
- Copy/share actions reuse the retained model and rendered Markdown rather than re-scraping X.
- Keep progress toasts during work; replace only the final transient success state.
- Add DOM tests covering clean capture, incomplete media, copy action, share failure after local
  success, keyboard dismissal, and accurate statistics.

## 4. Poll preservation

Polls are source content and currently represent a genuine fidelity gap.

### Model

Add a stable poll block containing only fields actually observed:

- Question, when distinguishable from surrounding post text
- Choice labels
- Visible percentages and/or vote counts
- Visible total vote count
- Poll status or remaining/closed text
- Source post ID and source URL
- A flag showing whether results were unavailable at capture time

Do not infer hidden results and do not convert missing data into zero.

### Extraction

- Put every X poll selector in `CONFIG.selectors` with fallbacks.
- Exclude polls belonging to nested quote cards from the outer post.
- Preserve quote poll attribution when a quoted post exposes poll data.
- If syndication data supplies more authoritative poll fields, prefer it only when mapping can be
  tested and attributed safely; DOM-only v1 support is acceptable.

### Rendering

- Offline HTML: accessible poll card with choices and observed results.
- Markdown: readable list/table that does not imply unavailable results.
- Manifest/debug output: structured poll content and completeness status.
- Thread and single-post paths must produce the same poll block contract.

### Tests

- Open poll with visible choices but hidden results
- Closed poll with percentages/vote count
- Poll inside a quote card
- Thread containing more than one poll
- No duplicate outer/quote poll attribution
